# Round 33 阶段验收与后三阶段实施设计

## 1. 目标

在不继续扩张功能范围的前提下，把当前版本从“主链路可用”推进到“可稳定验收、可稳定演示、可进入发布准备”。

本轮只按以下顺序推进：
1. 测试与发布准备
2. 管理端体验收口
3. 上传与预览稳定性增强

## 2. 设计边界

- 不新增分享外链、版本历史、协作编辑等超出 MVP 的能力。
- 不推翻当前方案 D 的页面结构，只做现有页面收口。
- 不将串行多文件上传直接改为并发上传，稳定性优先。
- 不引入新的基础设施依赖。

## 3. 第一阶段：测试与发布准备

### 3.1 目标
- 让当前版本具备一套可重复执行的验证流程。
- 明确“只启动后端即可访问页面”的交付方式。
- 收敛发布前检查项、回滚动作和验证命令。

### 3.2 实施内容
- 补一轮后端全量测试执行。
- 补前端关键 E2E，尤其是管理员页面与后端托管前端模式。
- 校正 `docs/SETUP.md` 中单后端托管前端的最终启动步骤。
- 校正 `docs/RELEASE_CHECKLIST.md` 和 `docs/ROLLBACK_PLAN.md`，把“应该检查什么”改成“如何检查、执行什么命令、预期结果是什么”。
- 如有必要，补最小 smoke 验证脚本说明。

### 3.3 验收标准
- 后端 `pytest -q` 可通过。
- 前端 `npm run build` 可通过。
- 前端 `npm run e2e` 可通过。
- 单后端托管模式下，`/files`、`/upload`、`/preview`、`/recycle`、`/admin/files`、`/admin/logs` 可访问。

## 4. 第二阶段：管理端体验收口

### 4.1 目标
- 让管理员使用“后台文件管理页”和“操作日志页”时，不再停留在开发态界面。

### 4.2 问题判断
当前管理端虽然功能可用，但存在这些典型问题：
- 筛选项风格与用户端不一致。
- 字段过于偏底层，例如直接暴露 `Owner ID`、`Target ID`，阅读成本高。
- 成功/失败/空态提示不统一。
- 表格布局与状态信息还不够精炼。

### 4.3 实施内容
- 文件管理页：
  - 将筛选区收敛为关键词、状态、用户标识等最小可用组合。
  - 将文件名、目录、用户、状态、时间拆分清楚。
  - 行操作反馈统一为页面消息或行内 loading。
- 日志页：
  - 将时间筛选改成更易输入的形式。
  - 提升日志详情可读性，避免一整段原始 JSON 直接堆叠。
  - 增强空态与错误态。
- 补充管理员页面 E2E 回归。

### 4.4 验收标准
- 管理页和日志页都具备清晰筛选、清晰结果、清晰空态。
- 管理员能独立完成查询、删除/恢复、日志筛选等核心操作。
- 管理端 E2E 可通过。

## 5. 第三阶段：上传与预览稳定性增强

### 5.1 目标
- 降低高频使用中最容易出现的不稳定感。

### 5.2 风险聚焦
- 多文件上传失败后无法单独重试。
- 长队列上传中用户无法取消。
- 预览失败时用户只看到报错，不清楚是权限问题、类型问题还是对象缺失。
- 历史数据兼容虽已有修复，但页面反馈仍不够明确。

### 5.3 实施内容
- 上传页：
  - 增加失败文件重试。
  - 增加队列取消或停止当前上传。
  - 统一成功、失败、取消三类状态文案。
- 预览页：
  - 对“不支持预览”“对象不存在”“权限不足”分别显示不同提示。
  - 保留右侧最近文件区，但增强失败后的切换体验。
- 补相关回归测试。

### 5.4 验收标准
- 多文件上传失败后可重试，不必整队列重来。
- 上传中可取消剩余队列。
- 预览失败提示可区分错误类型。
- 上传与预览回归测试通过。

## 7. 风险说明

- 这一轮不再扩功能，而是收口；若中途插入新需求，会直接打断发布准备节奏。
- 当前上传仍以串行为主，若中途改成并发，会显著增加测试和回归成本。
- 管理端与用户端使用同一套接口与对象存储数据，数据清理类动作仍需要谨慎对待。
