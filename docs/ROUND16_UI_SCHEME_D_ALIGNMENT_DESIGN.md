# ROUND16 UI 方案 D 对齐实施设计

## 1. 背景与目标
- 已确认 UI 基线为 V5 方案 D，原型路径：`docs/ui-prototypes/v5/scheme-d/preview.html`。
- 当前 `develop` 前端为功能优先骨架，未按方案 D 的页面结构与视觉令牌实现。
- 本轮目标：在不破坏已完成业务链路的前提下，将前端统一切换到方案 D 布局基线，并作为后续开发唯一 UI 基准。

## 2. 范围
- 仅对齐前端布局、路由结构、样式令牌与页面容器。
- 不变更后端接口契约，不新增复杂业务规则。
- 需要覆盖 7 个独立页面：登录、文件列表、上传、预览、回收站、后台管理、操作日志。

## 3. 方案 D 视觉与布局基线
- 颜色：`--bg #fafafa`、`--surface #fff`、`--line #e8e8e8`、`--text #111`、`--muted #6a6a6a`。
- 字体：标题 `Iowan Old Style/Songti SC`，正文 `Helvetica Neue/PingFang SC`。
- 结构：`topbar(78px) + shell(max-width:1040) + sidebar(nav) + main(kpi + content)`。
- 密度：中低密度，弱色块，强线条秩序。

## 4. 页面与路由对齐
- `/login`：独立登录页面。
- `/files`：文件列表主页面。
- `/upload`：上传独立页面（从列表页剥离上传区）。
- `/preview/:id?`：预览独立页面。
- `/recycle`：回收站页面。
- `/admin/files`：后台文件管理页面。
- `/admin/logs`：操作日志页面。

## 5. 组件边界
- 新增统一壳组件：`Topbar + Sidebar + PageMeta + KPI`。
- 业务页面只负责数据与操作，壳层负责布局与视觉一致性。
- 上传/预览从 `FileCenterView` 拆分为独立页面，避免单页承载过多职责。

## 6. 兼容与风险控制
- 所有 API 调用保持原路径与参数，避免后端联调回归风险。
- 先完成“结构对齐”，再逐步做“细节高保真还原”，防止大改导致不可验证。
- 每个最小提交必须满足 `npm run build` 可通过。

## 7. 最小提交拆分
- 提交 A（本提交）：设计文档与执行拆分。
- 提交 B：恢复 `docs/ui-prototypes/v5/scheme-*` 与 `docs/ui-previews/v5/*` 设计资产。
- 提交 C：落地方案 D 全局样式令牌与应用壳布局。
- 提交 D：路由对齐，新增上传页与预览页并接入现有业务能力。
- 提交 E：页面细节对齐与文档收口，固化“后续改动必须适配方案 D”的约束。

## 8. 验收标准
- 原型文件路径在当前分支可直接访问。
- 前端 7 个页面均可通过路由独立进入。
- 页面结构符合方案 D（顶部、导航、KPI、主内容区）。
- 构建通过：`cd frontend && npm run build`。
