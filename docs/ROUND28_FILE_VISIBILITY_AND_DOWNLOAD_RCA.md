# 第二十八轮开发：文件显示与下载异常 RCA

## 一、问题现象

1. 下载 `logs/readme.txt` 失败，提示“文件内容不存在”。
2. 页面把 `logs/readme.txt` 直接当成文件名显示，用户期望标题和文件名只显示 `readme.txt`。
3. 把 `logs` 重命名成 `log` 后，原本目录中的文件“不容易看到”。
4. 删除文件夹后，回收站里用户只注意到被删除的文件夹，没有清晰看到被删除的文件。
5. 文件树选中效果过弱，切换目录时不够明显。

## 二、可能原因

1. 下载失败：
   - 当前记录的 `object_key` 和真实对象存储路径可能发生历史不一致。
   - 现有下载 / 预览逻辑只按数据库中的 `object_key` 直接读取，一旦旧数据元路径不一致，就直接返回 404。

2. 文件名显示：
   - 前端直接把 `file_name` 路径字段当作文件名展示，没有拆分“文件名”和“所在目录”。

3. 重命名后文件不易发现：
   - 前端列表仍显示完整路径，重命名后路径变化使文件名信息被埋在路径里。
   - 文件树当前选中态不够强，用户不容易确认自己已切到新目录。

4. 回收站里“像是没有文件”：
   - 后端复现结果表明递归删除后，文件和文件夹都会进入回收站。
   - 但前端回收站把文件名显示成完整路径，识别成本高；同时文件和文件夹分成两个区域，文件并不够醒目。

5. 文件树选中态弱：
   - 当前样式只有轻量背景变化，没有明显边框/高亮/文字对比度增强。

## 三、验证步骤

1. 用后端 `TestClient` 复现完整链路：
   - 创建 `logs`
   - 上传 `logs/readme.txt`
   - 下载
   - 重命名 `logs -> log`
   - 再下载
   - 重命名 `log -> logs`
   - 递归删除 `logs`
   - 查看 `/recycle/files` 和 `/recycle/folders`

2. 检查前端文件列表、预览页、回收站页是否直接使用 `file_name` 路径字段做标题展示。

3. 检查文件树选中态 CSS 是否足够明显。

## 四、验证结果

1. 当前后端在“干净数据”下链路正常：
   - 上传后可下载
   - 重命名后仍可下载
   - 递归删除目录后，删除文件和删除文件夹都能进入回收站

2. 说明第 1 个问题更像是历史脏数据兼容问题：
   - 某些旧记录的 `object_key` 很可能与真实对象路径不一致
   - 当前代码缺少兼容性兜底查找

3. 第 2、3、4、5 个问题都属于前端展示和交互表达不足：
   - 路径与文件名未分离
   - 回收站文件信息不够直观
   - 目录切换状态不够清晰

## 五、最终根因

1. 下载异常根因：
   - 当前系统没有为历史错误元数据提供对象路径兼容查找机制。

2. 文件显示和“看不见”根因：
   - 前端把完整路径当作文件名展示，缺少 basename / directory 拆分层。

3. 回收站文件不醒目根因：
   - 文件数据存在，但展示层没有把“文件名”和“目录”分开，导致可读性差。

4. 文件树选中效果不明显根因：
   - 选中样式层级太弱。

## 六、修复方案

1. 后端：
   - 为下载 / 预览增加历史对象路径兼容查找。
   - 当 `object_key` 直接读取失败时，在同一上传会话目录下尝试按文件名做兼容查找。

2. 前端：
   - 文件列表、预览列表、回收站文件表统一拆分：
     - 文件名：只显示 basename
     - 所在目录：单独一列显示绝对路径
   - 下载默认文件名改为 basename，不再带目录前缀。
   - 增强目录树选中态样式。
   - 提升回收站文件区可读性和可识别性。
