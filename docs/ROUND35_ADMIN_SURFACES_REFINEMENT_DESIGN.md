# Round 35 管理端体验收口设计

## 1. 目标

在不扩展管理端功能边界的前提下，把现有两个管理员页面从“能用”收口到“可读、可筛、可回归验证”。

收口范围仅包含：
- `/admin/files`
- `/admin/logs`

不新增新的管理端页面，不引入 RBAC 细粒度权限。

## 2. 当前问题

### 2.1 后台文件管理页
- 筛选区仍带明显开发态字段名，例如 `Owner ID`。
- 文件名和目录路径混在一起，管理员难以快速判断文件归属位置。
- 缺少排序，面对多条记录时定位效率低。
- 成功反馈、空态和状态信息不够集中。

### 2.2 操作日志页
- 时间筛选要求手填 ISO，使用成本高。
- 日志详情直接展示原始 JSON，阅读负担大。
- 页面只有表格，没有摘要信息与空态引导。

## 3. 设计原则

- 保持现有 API 体系，不为了 UI 小问题重做后端结构。
- 优先提升管理员判断效率，而不是增加炫技交互。
- 能通过前端收口解决的问题，先不扩后端。
- 如确有必要，只补最小后端能力，例如统一排序参数。

## 4. 实施方案

### 4.1 后台文件管理页

#### 筛选区
- 保留：文件关键字、用户 ID、状态。
- 新增：排序方式。
- 文案收敛：`Owner ID` 改为 `用户 ID（可选）`。

#### 结果区
- 将 `file_name` 拆成：
  - 文件名
  - 所在目录
- `owner_id` 改为更易读的展示：`用户 #ID`。
- 增加管理摘要：
  - 当前结果数
  - 当前状态筛选
- 增加空态文案。

#### 交互
- 删除、恢复、彻底删除后维持静默刷新。
- 成功消息统一到页面顶部。

#### 后端配合
- `GET /admin/files` 增加 `sort_by` 参数。
- 复用现有文件列表排序规则。

### 4.2 操作日志页

#### 筛选区
- 保留：操作类型、用户 ID、开始时间、结束时间。
- 将开始/结束时间输入改为本地日期时间输入，不再要求用户手写 ISO。

#### 结果区
- 增加审计摘要：
  - 当前日志条数
  - 当前筛选动作
- `detail_json` 改为前端格式化摘要展示：
  - 优先显示 `file_name`、`range`、`size_bytes` 等关键字段
  - 无法解析时再显示原文
- 增加空态文案。

### 4.3 自动化验证
- 后端：补 `admin/files` 排序测试。
- 前端：新增管理员页面 E2E，覆盖：
  - 管理员查看文件
  - 管理员按关键字筛选
  - 管理员删除并恢复文件
  - 管理员筛选日志

## 5. 验收标准

- 管理员文件页能快速识别文件名、目录、用户、状态与操作结果。
- 日志页筛选无需手写 ISO 字符串。
- 日志详情可快速读懂关键字段。
- 管理端 E2E 可通过。
- 不影响现有用户端链路。
