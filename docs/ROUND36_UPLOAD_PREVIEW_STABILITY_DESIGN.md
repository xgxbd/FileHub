# Round 36 上传与预览稳定性增强设计

## 1. 目标

在不改变当前 MVP 主链路的前提下，解决上传与预览中最容易让用户感到“不稳定”的三类问题：
- 上传中无法停止队列
- 失败或中断后的文件无法直接重试
- 预览失败时提示过于笼统

## 2. 边界

- 不把当前串行上传直接改成并发上传。
- 不新增新的上传页面。
- 不扩展新的预览类型。
- 不引入新的状态管理库。

## 3. 当前问题

### 3.1 上传页
- 多文件队列可以顺序上传，但用户无法中途停止。
- 某个文件失败后，只能重新整批选择文件。
- 当前状态文案主要分为待上传、上传中、成功、失败，不够覆盖“已取消”。

### 3.2 预览页
- 预览失败时主要依赖后端原始错误文本，缺少更明确的前端语义提示。
- 用户难以区分：
  - 无权限
  - 文件内容缺失
  - 类型不支持

## 4. 实施方案

### 4.1 上传页

#### 队列控制
- 新增 `停止队列` 按钮。
- 点击后：
  - 若当前文件正在上传，则中断当前请求。
  - 当前文件标记为 `已取消`。
  - 后续未开始文件统一标记为 `已取消`。

#### 重试能力
- 新增 `重试未完成` 按钮。
- 范围包含：
  - `失败`
  - `已取消`
- 成功文件保持不变，不重复上传。

#### 状态定义
- `pending`：待上传
- `uploading`：上传中
- `success`：已完成
- `error`：失败
- `canceled`：已取消

#### 技术实现
- 前端上传请求接入 `AbortController`。
- 当前上传任务被取消时：
  - 中止当前分片请求或完成请求
  - 更新队列项状态
- 队列循环中检查 `cancelRequested`，避免继续处理后续文件。

### 4.2 预览页

#### 错误提示分流
- `400`：显示“当前类型不支持在线预览”
- `403`：显示“你无权预览该文件”
- `404` 且为对象缺失：显示“文件元数据存在，但文件内容不存在，请重新上传”
- 其他错误：显示原始错误或兜底文案“预览失败”

#### 技术实现
- `frontend/src/api/files.js` 为预览请求抛出带 `status` 的错误对象。
- `PreviewView.vue` 按状态码映射用户可理解的提示。

## 5. 自动化验证

- 新增上传队列控制 E2E：
  - 选择多个文件
  - 开始上传
  - 中途停止
  - 重试未完成
  - 最终完成
- 保留现有上传预览 E2E，确保不回归。

## 6. 验收标准

- 多文件上传中可停止剩余队列。
- 已取消或失败的文件可直接重试，无需重新选文件。
- 预览失败提示能区分主要失败类型。
- 常规 E2E 继续通过。
